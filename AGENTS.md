# AGENTS.md

Инструкция для AI-агентов и новых исполнителей, которые работают в этом репозитории.

## Главный принцип

Этот репозиторий - не универсальный продукт и не учебная платформа "для всех". Это рабочая система boutique-проекта Mentor Analytics. По умолчанию выбирай решения, которые:

- уменьшают нагрузку на основателя
- повышают понятность материалов и процессов
- не требуют лишней операционки
- не превращают проект в "онлайн-школу"

## Правила работы по разделам

### `content/`

- Не перескакивай сразу к полному продакшн-пакету, если по workflow нужен сначала бриф
- Для YouTube сначала делай `brief.md`, потом `script.md`, затем production-артефакты
- Учитывай ограничения платформ, особенно для VK и Instagram-упоминаний

### `courses/`

- Не создавай финальный `guide.md`, пока структура и разделы не согласованы
- Сохраняй практичный тон: 20% теории, 80% пользы
- Если нужен новый раздел курса, соблюдай существующий стиль именования и структуры

### `resume/`

- Сначала анализ, потом финальные артефакты
- Не придумывай "рыночные факты" без опоры на уже сохраненные материалы проекта
- Сохраняй прикладной формат: скоринг, шаблоны формулировок, конкретные правки

### `marketing/`

- Работай осторожно: здесь могут лежать ТЗ, планы и полуфинальные документы
- Если структура задачи не очевидна, сначала изучи соседние файлы и только потом вноси изменения

### `scripts/`

- Перед изменением скриптов проверь, какие файлы они читают и куда пишут результат
- Не меняй путь к `.secrets/` и рабочим директориям без явной причины
- Любые действия с YouTube OAuth, Google Sheets и внешними API считай чувствительными

## Что можно делать без отдельного согласования

- Улучшать структуру документации
- Добавлять README, инструкции, шаблоны, если они не ломают текущий workflow
- Уточнять naming conventions и порядок запуска локальных скриптов
- Создавать безопасные вспомогательные файлы вроде `requirements.txt`

## Что лучше сначала проверить или явно отметить

- Любые изменения в бизнес-правилах, ценах, unit economics и позиционировании
- Любые автоматизации, которые публикуют что-то наружу
- Любые правки, которые затрагивают `.secrets/`, OAuth или интеграции
- Любые действия, которые могут конфликтовать с незакоммиченными пользовательскими изменениями

## Naming conventions

- Новые Markdown-файлы называй в стиле существующего раздела
- Для контента с датой предпочитай формат `YYYY-MM-DD_slug.md`
- Для production-пакетов YouTube используй короткий осмысленный slug темы
- Не переименовывай существующие файлы ради "красоты", если это не часть явной задачи

## Ожидания от результата

Хороший результат в этом репозитории:

- легко найти
- можно использовать без дополнительной расшифровки
- не противоречит существующему workflow
- усиливает текущую систему, а не создаёт параллельную
