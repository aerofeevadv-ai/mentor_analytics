# Гайд: Claude Code для аналитика данных

Практическое руководство по настройке Claude Code для продуктовой / бизнес-аналитики.
С примерами из реальных проектов.

---

## Что такое Claude Code

Claude Code — это **агент** от Anthropic, который работает в терминале (Terminal на Mac, Windows Terminal на Windows — программа с текстовым интерфейсом). Не плагин к редактору, не чат-бот в браузере, а автономный инструмент, который:

- Читает и пишет файлы в твоём проекте
- Запускает команды в терминале
- Запоминает контекст между сессиями
- Работает с Git, SQL-файлами, отчётами

Для аналитика это значит: ты описал свои таблицы, метрики и правила один раз — и Claude знает их в каждой сессии. Не надо каждый раз копировать «у меня PostgreSQL, таблица orders, поле amount включает НДС».

---

## Чем Claude Code отличается от веб-чатов

| Боль аналитика | Веб-чат (ChatGPT / Claude) | Claude Code |
|---|---|---|
| Каждый раз объясняю контекст | Копирую описание таблиц в чат | Контекст загружается автоматически из `CLAUDE.md` и `knowledge/` |
| Ошибки в именах колонок | Claude выдумывает имена | Схемы таблиц лежат в файлах — Claude читает их |
| Потерял наработки прошлой сессии | Начинаю с нуля | Memory хранит находки между сессиями |
| Копирую SQL туда-сюда | Копирую из чата в терминал и обратно | Claude пишет SQL прямо в файл, читает результаты |
| Нет самопроверки | Надеюсь что запрос верный | Автоматические sanity checks и `/review` |
| Однотипные задачи | Каждый раз заново | Шаблоны задач и команды (`/new-task`) |

### Главная разница — не «умнее», а «видит больше»

Модель Claude в веб-чате и в Claude Code — **одна и та же**. Разница не в интеллекте, а в том, что Claude Code:

1. **Видит весь проект** — все файлы, структуру папок, схемы таблиц
2. **Помнит контекст** — `CLAUDE.md` загружается в каждую сессию, Memory хранит выводы между сессиями
3. **Действует** — пишет файлы, запускает команды, создаёт папки
4. **Проверяет себя** — можно настроить автоматические sanity checks

В веб-чате ты — курьер между Claude и реальностью. В Claude Code он работает с реальностью напрямую.

## Аналоги: что ещё есть на рынке

### Сравнение инструментов

> IDE (Integrated Development Environment) — программа для написания кода, как VS Code. API — программный интерфейс, через который сервисы общаются друг с другом.

| | Claude Code | Cursor | Google Antigravity | Qwen Code | DeepSeek |
|---|---|---|---|---|---|
| **Тип** | Агент (терминал) | IDE (редактор кода) | Веб-IDE с агентами | Плагин для IDE | Веб-чат + API |
| **Как работает** | Даёшь задачу → делает сам | Пишешь код → AI помогает | Несколько агентов параллельно | Автодополнение + чат в IDE | API: подключаешь к любому IDE |
| **Контекст проекта** | `CLAUDE.md` + `knowledge/` + Memory | `.cursorrules` + индекс кодовой базы | Workspace с агентами | Контекст через IDE-плагин (128K) | Кэширование через API |
| **Лучше всего для** | Сложные задачи, много файлов, автономная работа | Ежедневная работа с кодом в IDE | Мультиагентные workflow | Бюджетный AI-кодинг | Максимальная экономия на токенах |
| **Цена** | $20/мес (Pro) или $100/мес (Max) | $20/мес (Pro) | Бесплатно (бета) + кредиты | ~$10–50/мес | $0.03–0.42 за 1M токенов |
| **Модели** | Claude (Opus, Sonnet, Haiku) | Claude, GPT-4o, собственные | Gemini, Claude, GPT | Qwen 3.5, Qwen 3-Coder | DeepSeek V3.2 (Chat + Reasoner) |
| **Для аналитика** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ | ⭐⭐ |

### Cursor — главный конкурент

Cursor — форк VS Code со встроенным AI. Самый популярный AI-редактор кода. Выглядит как обычный VS Code, но с AI-автодополнением, чатом и агентным режимом.

**Плюсы для аналитика:**
- Привычный интерфейс (если уже работал в VS Code)
- AI-автодополнение прямо в редакторе — подсказывает SQL на лету
- Поддержка нескольких моделей (Claude, GPT-4o)
- `.cursorrules` — аналог `CLAUDE.md` для настройки контекста

**Минусы:**
- Нет Memory между сессиями (только в рамках одного чата)
- Контекст проекта ограничен — не всегда подхватывает нужные файлы
- Требует больше ручного управления: «посмотри файл X», «используй таблицу Y»

**Вердикт:** Cursor хорош для тех, кто уже пишет код в VS Code. Claude Code — лучше для аналитиков, которые работают с задачами, а не с редактором.

### Google Antigravity — мультиагентная веб-IDE

Antigravity — AI-среда разработки от Google. Работает в браузере: ты даёшь задачу, и несколько агентов выполняют её параллельно. Есть встроенный браузер для QA и интерфейс «Mission Control» для управления агентами.

**Плюсы:** бесплатная бета, поддержка разных моделей (Gemini, Claude, GPT), параллельное выполнение задач.
**Минусы:** молодой продукт, слабая документация, непредсказуемое ценообразование (уже были протесты пользователей по лимитам). Не заточен под аналитику — скорее для разработчиков.

### Qwen Code — бюджетный AI-ассистент от Alibaba

Qwen Code — плагин для IDE (работает с Cursor, VS Code, Cline). Модели Qwen 3-Coder используются как альтернативные AI-модели внутри этих редакторов — вместо Claude или GPT. Качество приличное за цену в 3–5 раз ниже конкурентов. Контекстное окно 128K токенов.

**Плюсы:** $10/мес за Lite-план (~90K запросов/мес), работает с разными IDE, хорош для рутинного SQL.
**Минусы:** менее известен на российском рынке, качество ниже Claude на сложных задачах, документация на английском.

### DeepSeek — самый дешёвый API

DeepSeek V3.2 — модель с двумя режимами: **Chat** (быстрые ответы, простые задачи) и **Reasoner** (цепочка рассуждений для сложной логики и кода). Есть бесплатный веб-чат на сайте DeepSeek — можно использовать прямо сейчас. Для встраивания в редактор кода нужна техническая настройка.

**Плюсы:** самый дешёвый вариант, режим Reasoner хорошо справляется с логическими задачами, бесплатный веб-чат.
**Минусы:** чтобы использовать как полноценный рабочий инструмент — нужно самому подключать к редактору. Нет Memory, нет `CLAUDE.md`. Для аналитика — скорее дополнение, чем основной инструмент.

### Что выбрать аналитику?

**Короткий ответ:** Claude Code.

**Почему:**
1. **Memory** — единственный инструмент, который помнит контекст между сессиями
2. **`CLAUDE.md` + `knowledge/`** — самая гибкая система управления контекстом
3. **Автономность** — даёшь задачу целиком, получаешь результат. Не надо «пилотировать» AI на каждом шаге
4. **Лучшее качество кода** — Claude Opus — самая мощная модель, пишет наиболее качественный код
5. **Терминал** — не нужен тяжёлый IDE, работает везде

**Если бюджет ограничен:** начни с бесплатного веб-чата DeepSeek или Cursor (бесплатный tier) → когда поймёшь ценность AI-ассистента → переходи на Claude Code.

## Архитектура: из чего состоит настройка

> Не пытайся запомнить всё сразу — это итоговая картинка. Ниже разберём каждый элемент по шагам.

```
my-analytics-project/
├── CLAUDE.md                  # Главный конфиг: роль, правила, workflow
├── knowledge/                 # База знаний: таблицы, метрики, методология
│   ├── data-sources.md        # Какие таблицы есть, когда какую использовать
│   ├── table-schema-X.md      # Схема конкретной таблицы (поля, типы, описания)
│   ├── glossary.md            # Термины и аббревиатуры компании
│   ├── sql-pitfalls.md        # Частые ошибки в SQL
│   └── methodology-X.md       # Методология (AB-тесты, воронки, когорты...)
├── .claude/
│   └── commands/              # Кастомные команды (/review, /new-task)
│       └── review.md
└── tasks/                     # Папки задач (одна задача = одна папка)
    └── TASK-123/
        ├── task.md            # Постановка
        ├── queries/           # SQL-файлы
        └── report.md          # Результат
```

**Что такое `knowledge/`?** Это просто папка с текстовыми файлами (.md). В них ты описываешь таблицы, термины, правила. Claude читает эти файлы, когда ему нужна информация. Ничего волшебного — обычные текстовые файлы, которые ты (или Claude по твоей просьбе) создаёшь и обновляешь.

### Что загружается автоматически каждую сессию

- `CLAUDE.md` (всегда, полностью) — держи коротким, до 200 строк
- Memory (Claude создаёт и пишет туда сам, когда ты говоришь «запомни: ...»)

### Что Claude читает по запросу

- Файлы из `knowledge/` — когда пишет SQL или анализирует данные
- Файлы задач — когда работает над конкретной задачей

## Когда тебе это пригодится

Этот гайд — для работы на реальном проекте. Когда выйдешь на работу аналитиком — настроишь Claude Code под свою компанию и базы данных. Пока учишься — можешь использовать Claude Code для учебных задач и подготовки к собеседованиям:

```
Ты: "Я готовлюсь к собеседованию на аналитика.
     Дай мне задачу на SQL: подсчитать retention по когортам.
     Опиши таблицы, напиши условие и покажи эталонный запрос."

Ты: "Объясни мне оконные функции на примерах.
     Покажи ROW_NUMBER, RANK и LAG — когда что использовать."
```

> 📌 Если ещё не установил Claude Code — перейди в раздел «Установка и начало работы» в конце гайда, потом возвращайся сюда.

## Сценарий 1: Старт с нуля

> Ты пришёл в новую компанию. Документации нет. Есть доступ к базе данных и Jira.

### Шаг 1. Открой Claude Code и расскажи ему всё что знаешь

Создай папку для проекта, открой в ней Claude Code и просто расскажи ему о себе:

```
Ты: "Привет! Я продуктовый аналитик в компании [Название].
     Домен — маркетплейс недвижимости. SQL — PostgreSQL.
     Основные таблицы: orders, users, events.
     Создай мне CLAUDE.md с этой информацией."
```

Claude сам создаст файл, предложит структуру и спросит, если чего-то не хватает. Не пытайся описать всё сразу — `CLAUDE.md` растёт органически по мере работы.

### Шаг 2. Начни работать — knowledge появится сам

Первая задача — лучший способ наполнить knowledge base.

```
Ты: "Нужно посчитать конверсию из регистрации в первую покупку за последний месяц"
Claude: "Какие таблицы содержат регистрации и покупки?"
Ты: "Регистрации в schema.users, покупки в schema.orders"
Claude: [пишет SQL]
```

После первой задачи попроси Claude сохранить знания:

```
Ты: "Запиши схему таблиц users и orders в knowledge/"
```

Если ты не помнишь точные поля таблиц — попроси Claude помочь:

```
Ты: "Напиши мне SQL-запрос, чтобы увидеть структуру таблицы users —
     все колонки, типы данных, nullable"
```

Claude напишет нужный запрос под твой диалект SQL. Ты запустишь его в своём редакторе, скинешь результат обратно — и Claude сохранит схему в `knowledge/tables-users.md`.

### Шаг 3. После 3–5 задач — попроси Claude навести порядок

К этому моменту ты уже знаешь, какие таблицы используешь чаще всего, какие ошибки Claude допускает, какие термины приходится объяснять. Скажи ему:

```
Ты: "Создай knowledge/glossary.md — выпиши туда все термины компании,
     которые я описывал. И knowledge/sql-pitfalls.md — типичные ошибки
     при работе с нашими таблицами."
```

> 📌 Claude помнит контекст только внутри одной сессии. Между сессиями помогают `CLAUDE.md`, `knowledge/` и Memory. Поэтому важно фиксировать находки в файлы, а не держать в голове.

### Шаг 4. После 10+ задач — добавь автоматизацию

Скажи Claude:

```
Ты: "Создай шаблон задачи в tasks/_template/ и команду /review
     для самопроверки SQL"
```

Claude создаст нужные файлы и объяснит, как ими пользоваться.

## Сценарий 2: Документация уже есть

> В компании есть Confluence, описания таблиц, методологии. Нужно подключить к Claude Code.

### Шаг 1. Перенеси ключевое в проект

**Вариант А (простой):** Сохрани документ к себе на ноутбук (PDF, TXT, скопировать текст) и попроси Claude создать из него knowledge-файл:

```
Ты: "Вот описание таблицы orders из Confluence [вставляешь текст].
     Создай из этого knowledge/tables-orders.md в нашем формате."
```

**Вариант Б (продвинутый):** Claude Code умеет подключаться к внешним сервисам (Confluence, Google Docs) через специальные плагины — MCP-серверы. После настройки Claude сам читает документацию без копирования. Настройка требует опыта — погугли «Claude Code MCP Confluence», есть готовые решения от Atlassian и сообщества.

> 📌 Для старта хватит варианта А. MCP — когда документации много и она часто обновляется.

**Что переносить в первую очередь (по приоритету):**

| Приоритет | Что | Куда | Зачем |
|---|---|---|---|
| 1 | Схемы топ-10 таблиц | `knowledge/tables-*.md` | 80% ошибок Claude — неправильные имена колонок |
| 2 | Глоссарий (термины бизнеса) | `knowledge/glossary.md` | Claude путает ваши термины с общеупотребительными |
| 3 | Карта источников данных | `knowledge/data-sources.md` | «Когда какую таблицу использовать» |
| 4 | Методология (AB / воронки / когорты) | `knowledge/methodology-*.md` | Чтобы Claude не выдумывал свою методологию |
| 5 | Частые SQL-паттерны | `knowledge/sql-patterns.md` | Проверенные конструкции вместо изобретения с нуля |

### Формат файла схемы таблицы (пример)

```markdown
# schema.orders — Заказы

**Гранулярность:** 1 строка = 1 заказ
**Основной ключ:** order_id
**Партиция:** created_date (поле, по которому таблица разбита на части для ускорения запросов)

| Поле | Тип | Описание |
|------|-----|----------|
| order_id | bigint | ID заказа |
| user_id | bigint | ID пользователя |
| created_at | timestamp | Дата создания |
| status | varchar | Статус: 'completed', 'cancelled', 'pending' |
| amount | decimal(10,2) | Сумма заказа (с НДС) |
| region_id | int | ID региона (ссылка на таблицу regions) |

**Важно:**
- `amount` включает НДС. Для суммы без НДС: `amount / 1.2`
- Отменённые заказы (`status = 'cancelled'`) остаются в таблице
- Дубли возможны при повторной оплате — дедупликация по `order_id`
```

Секция «Важно» — ключевая. Именно там описаны ловушки, которые Claude не может узнать из типов колонок.

### Шаг 2. Создай CLAUDE.md

Возьми шаблон из раздела «Быстрый старт» в конце этого гайда и попроси Claude адаптировать его:

```
Ты: "Вот шаблон CLAUDE.md [вставляешь шаблон из раздела 'Быстрый старт'].
     Адаптируй под нашу компанию: ClickHouse, таблицы prod.orders,
     prod.events, prod.users. Даты в UTC, отчёты в MSK.
     user_id ≠ device_id — это важно."
```

Claude создаст файл и может задать уточняющие вопросы.

### Шаг 3. Перенеси SQL-паттерны

У каждой компании есть свои идиомы: как считать MAU, как определять новых пользователей, как фильтровать ботов. Попроси Claude собрать их:

```
Ты: "Создай knowledge/sql-patterns.md. Запиши туда наши проверенные
     SQL-паттерны: как мы считаем новых пользователей, как исключаем
     ботов, как считаем revenue без НДС."
```

Claude создаст файл с нужными SQL-конструкциями. Проверь и поправь — это будет эталон для будущих задач.

## Сценарий 3: Зрелая настройка — оптимизация

> У тебя уже есть `CLAUDE.md` и `knowledge/`. Как выжать максимум.

### 3.1. Разгрузи CLAUDE.md

`CLAUDE.md` загружается **каждый раз**. Если он раздулся до 300+ строк:

**Правило:** В `CLAUDE.md` — только правила поведения и навигация. Детали — в `knowledge/`.

| В CLAUDE.md (всегда загружается) | В knowledge/ (читается по запросу) |
|---|---|
| «Используй LEFT JOIN по умолчанию» | Полная схема таблицы с 50 полями |
| Таблица: что → какая таблица | Методология AB-тестов на 80 строк |
| «α = 0.01 для AB-тестов» | Чеклист SQL-ошибок из 20 пунктов |

### 3.2. Добавь кастомные команды

#### Что такое /review

**Review** — это самопроверка задачи перед отправкой результата. Claude перечитывает постановку, SQL, выводы и ищет ошибки — как второй аналитик, который смотрит свежим взглядом.

#### Что такое sanity check

**Sanity check** — «проверка на здравый смысл». Дополнительные SQL-запросы, которые проверяют основной результат: нет ли дублей после JOIN, нет ли пропущенных дней в данных, порядок величин совпадает с ожиданием. Это страховка от ситуации «число красивое, но неправильное».

Пример: основной запрос дал 50K строк. Проверочный `SELECT COUNT(DISTINCT order_id)` дал 45K. Значит 5K строк — дубли после JOIN, и нужно разобраться почему.

#### Как запускать

Команды можно вызывать двумя способами — результат одинаковый:

```
/review PA-20098          ← через слэш-команду
Проведи ревью PA-20098   ← обычным текстом
```

> 📌 Слэш-команды — это просто шаблоны промптов. Claude понимает и обычный текст. Используй то, что удобнее.

#### Настройка команды /review

Файл `.claude/commands/review.md`:

```markdown
Проведи ревью задачи $ARGUMENTS:
(Claude подставит сюда то, что ты напишешь после /review.
Например: /review PA-20098 → $ARGUMENTS = PA-20098)

1. Найди папку задачи в tasks/ или tasks/ab/
2. Прочитай task.md, все SQL из queries/, report_for_me.md
3. Прочитай knowledge/sql-pitfalls.md и knowledge/data-sources.md
4. Проверь:
   - Бизнес-логика: метрики соответствуют цели задачи?
   - SQL: правильные таблицы, JOIN-ы, фильтры, агрегация?
   - Выводы подкреплены данными?
5. Выдай вердикт: Ок / Замечания / Критические ошибки
```

#### Настройка команды /new-task

Файл `.claude/commands/new-task.md`:

```markdown
Создай новую аналитическую задачу $ARGUMENTS:

1. Определи номер тикета из аргумента (например, PA-20100)
2. Спроси тип задачи: ad hoc или AB-тест
3. Создай папку tasks/[PA-XXXXX]/ (или tasks/ab/[PA-XXXXX]/ для AB)
4. Скопируй содержимое из tasks/_template/ (или tasks/ab/_template/)
5. Замени PA-XXXXX на реальный номер тикета
6. Создай подпапки queries/ и results/
```

### 3.3. Используй Memory для накопления опыта

Memory — это долговременная память Claude между сессиями. Claude создаёт и обновляет её сам — тебе не нужно создавать файлы вручную. Просто скажи «запомни: ...» — и в следующей сессии Claude это будет знать.

**Что стоит запоминать:**

```
"Запомни: в таблице orders поле region_id — varchar,
не bigint. При JOIN с regions нужен CAST."

"Запомни: для AB-тестов мы используем α = 0.01, не 0.05."

"Запомни: таблица events за декабрь содержит дубли —
всегда дедуплицируй по event_id."
```

**Что НЕ стоит запоминать:**

- Текущий статус задачи (это в `task.md`)
- Временные решения («пока используем костыль X»)
- Информацию, которая уже в `knowledge/`

### 3.4. Настрой автоматические проверки в CLAUDE.md

```markdown
## Самоконтроль (Claude делает автоматически)

### Перед написанием SQL
- Прочитай knowledge/sql-pitfalls.md
- Определи и обоснуй тип каждого JOIN

### Sanity checks — автоматически для сложных запросов
Если запрос содержит 2+ JOIN, сложную классификацию или агрегацию
3+ полей — приложи проверочные запросы:
- COUNT(*) vs COUNT(DISTINCT id) — нет ли дублей после JOIN
- COUNT(DISTINCT date) — нет ли пропущенных дней
- COUNT NULL — сколько строк не нашли пару после LEFT JOIN

### Сверка результатов
- Если результат > 2x от ожидания — скажи сам, не молчи
```

## Паттерны эффективной работы

### Паттерн 1: «Инициатива на Claude»

**Плохо:**

```
Ты: "Напиши SQL для подсчёта заказов по категориям за январь"
Claude: [пишет SQL]
```

**Хорошо:**

```
Ты: "Задача: посчитать эффективность рекламных кампаний.
     Вот постановка из Jira: [текст]"
Claude: "Беру данные из marketing_events (потому что нужны utm-метки).
        Предлагаю план:
        1. Посчитать заказы по рекламным кампаниям
        2. Разделить на первичные / повторные
        3. Посчитать конверсию и CPA по каждой кампании
        Ок?"
```

Для этого в `CLAUDE.md`:

```markdown
## Рабочий процесс
Claude берёт инициативу:
- Сам выбирает источник данных и объясняет почему
- Предлагает гипотезы и план анализа
- Если неоднозначно — спрашивает подтверждение
```

> 📌 Claude предлагает план и пишет SQL, но запускаешь запросы в базе данных ты. Claude не подключается к твоей БД напрямую.

### Паттерн 2: «Резюме перед запуском»

После каждого сложного SQL Claude пишет в чат:

```
«Беру таблицу orders, фильтрую status = 'completed' за январь,
джойню с users по user_id через LEFT JOIN (хотим видеть заказы
без привязки к профилю), считаю revenue и количество заказов.
Допущение: не исключаю повторные заказы.
Ок?»
```

Ты проверяешь логику **на уровне смысла**, не читая SQL.

### Паттерн 3: «report_for_me.md — разбор для ревью»

Параллельно с SQL Claude ведёт файл `report_for_me.md`:

```markdown
# TASK-100 — Разбор логики

## Что считаем и зачем
Эффективность рекламных кампаний: сколько заказов приносит
каждая кампания и какая конверсия из клика в покупку.

## Логика расчёта
1. Берём события из marketing_events за январь 2026
2. Парсим utm_source / utm_campaign для привязки к кампаниям
3. Джойним с orders по user_id (LEFT JOIN — хотим видеть клики без покупок)
4. Считаем: клики, заказы, конверсия, CPA

## Допущения
- Берём только completed-заказы (status = 'completed')
- Атрибуция last-click (последний клик перед заказом)

## Sanity checks
| Проверка | Ожидание | Факт |
|---|---|---|
| Всего заказов за январь | ~10K | ? |
| Конверсия из клика | 2–5% | ? |
```

Этот файл — твоя страховка. Перед отправкой результата тимлиду ты читаешь его за 2 минуты и понимаешь всю логику.

### Паттерн 4: «Не молчи о подозрительном»

В `CLAUDE.md`:

```markdown
- Если результат расходится с ожиданием > 2x — скажи сам, не жди вопроса
- Если видишь подозрительные NULL или нули — предупреди
```

Claude начнёт говорить:

```
"Результат: 500K заказов за январь. В task.md ожидание ~50K.
Расхождение в 10x. Возможные причины:
1. Не отфильтрованы отменённые заказы
2. JOIN размножил строки (дубли)
Запускаю проверку на дубли..."
```

### Паттерн 5: «Аудит проекта»

Попроси Claude посмотреть на проект целиком и найти пробелы:

```
Ты: "Проведи аудит моего проекта. Посмотри структуру папок,
     CLAUDE.md, файлы в knowledge/. Что не хватает?
     Какие таблицы упомянуты в задачах, но не описаны?
     Есть ли устаревшая информация?"
```

Claude пройдётся по файлам и выдаст конкретные рекомендации:

```
Claude: "Нашёл 3 проблемы:
1. Таблица prod.campaigns упоминается в tasks/PA-20095,
   но нет knowledge/tables-campaigns.md
2. В CLAUDE.md указан диалект Presto, но в последних 5 задачах
   все запросы на ClickHouse — обновить?
3. knowledge/glossary.md не обновлялся 2 месяца —
   добавить термины из последних задач?"
```

Полезно делать аудит раз в месяц или при смене проекта/команды.

### Паттерн 6: «Claude ошибся — что делать»

Claude не идеален. Он может написать неверный SQL, перепутать колонки, дать неправильный вывод. Это нормально. Что делать:

**Если результат неверный:**
```
Ты: "Этот запрос даёт неправильный результат: revenue отрицательный,
     такого быть не может. Перепроверь логику расчёта."
```

**Если Claude использует несуществующую колонку:**
```
Ты: "Колонки payment_date нет в таблице orders. Есть created_at
     и completed_at. Перепиши запрос."
```

**Если вывод не совпадает с реальностью:**
```
Ты: "Ты написал что конверсия 50% — это нормально. Нет, для нашего
     продукта норма 2-5%. Пересмотри выводы."
```

Главное — объясни *что* не так и дай правильный контекст. Чем конкретнее — тем лучше Claude исправится.

## Антипаттерны: чего НЕ делать

### 1. Не пиши всё в CLAUDE.md

`CLAUDE.md` загружается каждый раз. 500 строк = лишние токены в каждом сообщении.
**Правило:** `CLAUDE.md` < 200 строк. Остальное — в `knowledge/`.

### 2. Не дублируй информацию

Если схема таблицы описана в `knowledge/tables-orders.md`, не копируй её в `CLAUDE.md`.
Дублирование → рассинхрон → ошибки.

### 3. Не надейся на память Claude между сессиями без Memory

Без `CLAUDE.md` и `knowledge/` каждая сессия начинается с нуля.
Без Memory — Claude не помнит, что «в прошлый раз мы обнаружили дубли в таблице X».

### 4. Не описывай «что делать» — описывай «как думать»

**Плохо:** «Всегда используй LEFT JOIN»

**Хорошо:** «Выбери тип JOIN и обоснуй: LEFT если хотим сохранить все строки левой таблицы, INNER если нужны только совпадения»

### 5. Не пропускай sanity checks

Самая частая ошибка: запустил SQL, получил число, вставил в отчёт.
**Всегда:** Claude выдаёт основной запрос + проверочные запросы. Запускай оба.

## Ограничения: что нужно знать

1. **Claude может ошибаться в именах колонок.** Если таблица не описана в `knowledge/` — Claude будет угадывать имена полей. Иногда угадывает верно, иногда нет. Поэтому `knowledge/` так важен.

2. **Данные уходят на серверы Anthropic.** Всё, что ты пишешь в Claude Code — промпты, SQL, результаты запросов — обрабатывается на серверах Anthropic. Не вставляй персональные данные клиентов, пароли и секретные ключи. SQL-запросы и агрегированные числа — ок.

3. **Нужен интернет.** Claude Code работает через API — без интернета не запустится.

4. **Стоит денег.** Подписка Pro ($20/мес) или Max ($100/мес). На Pro хватает для 5–10 задач в день. На Max — без ограничений.

```bash
npm install -g @anthropic-ai/claude-code

# Перейди в папку проекта
cd my-analytics

# Запусти
claude
```

При первом запуске Claude попросит авторизоваться через аккаунт Anthropic (нужна подписка Pro за $20/мес или Max за $100/мес).

### Первый запуск

После запуска `claude` в терминале появится строка ввода — просто пиши текст, как в чате:

```
Привет! Давай настроим проект для аналитики.
Создай CLAUDE.md по шаблону для продуктового аналитика.
```

### Полезные команды

| Команда | Что делает |
|---|---|
| `claude` | Запустить Claude Code |
| `/help` | Список встроенных команд |
| `/review` | Кастомная команда (если настроена) |
| `/new-task PA-123` | Кастомная команда (если настроена) |
| `Ctrl+C` | Прервать текущее действие |
| `/clear` | Очистить контекст текущей сессии |
| `/cost` | Сколько токенов потрачено в этой сессии |
| `/compact` | Сжать контекст — освободить место, сохранив ключевое |

### Контекст и когда перезапускать сессию

**Что такое контекст.** Каждая сессия Claude Code имеет «окно контекста» — ~200 000 токенов (токен — единица текста для AI, примерно 0.5–1 слово; 200K токенов ≈ 100 000 русских слов). Сюда входит всё: твои сообщения, файлы которые Claude читал, SQL который писал, результаты. Когда контекст заполняется — Claude начинает забывать ранние детали.

**Как следить.** Набери `/cost` — увидишь сколько токенов использовано. Claude покажет статус-бар с заполненностью контекста.

**Когда перезапускать:**

| Ситуация | Что делать |
|---|---|
| Контекст заполнен на 70%+ | `/compact` или начни новую сессию |
| Переключаешься на другую задачу | Лучше новая сессия — старый контекст будет мешать |
| Claude начал «забывать» что обсуждали раньше | `/compact` или новая сессия |
| Быстрый вопрос, не связанный с текущей задачей | Новая сессия — не засоряй контекст |

**`/compact` vs новая сессия:**
- `/compact` — Claude сжимает историю, оставляя ключевые решения и код. Удобно, если задача ещё в процессе.
- Новая сессия — чистый старт. `CLAUDE.md` и Memory подгрузятся автоматически, контекст задачи — нет. Лучше для новой задачи.

> 📌 Частая ошибка: работать в одной сессии весь день. После 3–4 задач контекст переполнен, Claude путает детали разных задач. Одна сессия = одна задача.

## Быстрый старт: скопируй и адаптируй

Минимальный `CLAUDE.md` для аналитика — скопируй и замени `[...]`:

```markdown
# CLAUDE.md

## Роль
Ты — [продуктовый/бизнес/маркетинг] аналитик в [Компания], [домен].
Общайся по-русски, коротко, по делу.

## SQL
- Диалект: [PostgreSQL / ClickHouse / Presto / BigQuery]
- Если запрос > 50 строк — спроси разрешение

## Рабочий процесс
1. Я скидываю задачу
2. Ты сам выбираешь источник данных из knowledge/ и предлагаешь план
3. Я подтверждаю — ты пишешь SQL
4. Я запускаю запрос и скидываю результат
5. Ты делаешь выводы

## Ключевые инварианты
- [user_id ≠ device_id / другие важные различия]
- [Даты в UTC/MSK]
- [Валюта с НДС / без]

## Источники данных
> Схемы: knowledge/tables-*.md

| Таблица | Описание | Когда использовать |
|---------|----------|-------------------|
| [schema.orders] | [Заказы] | [Revenue, конверсии] |
| [schema.events] | [Клик-стрим] | [Воронки, поведение] |

## Самоконтроль
- Перед сложным SQL — прочитай knowledge/sql-pitfalls.md
- Для запросов с 2+ JOIN — приложи sanity checks
- Если результат > 2x от ожидания — скажи сам
```

Дальше — расти `knowledge/` по мере работы. Каждая задача = повод добавить схему таблицы или паттерн.
